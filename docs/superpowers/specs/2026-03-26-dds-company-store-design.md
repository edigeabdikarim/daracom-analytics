# Spec: Отображение статей «Компания» в Сводной матрице ДДС

**Дата:** 2026-03-26
**Файл:** `dds.html` (только)
**GAS:** не затрагивается

---

## Контекст

В данных ДДС появилось новое значение поля `store` — `"Компания"`. Оно означает, что операция относится ко всей компании целиком и не может быть привязана к конкретному магазину. По условию данных, статьи с `store = "Компания"` **не имеют транзакций в других магазинах** (взаимоисключающее условие, гарантировано на уровне данных).

Текущая проблема: `"Компания"` попадает в массив `stores` и отображается как обычная колонка в матрице статей × магазины, из-за чего все магазинные ячейки у таких статей — нули, а значение сосредоточено в одной колонке «Компания». Это визуально запутывает.

---

## Решение

### Поведение в матрице

Для статей, у которых **все транзакции** имеют `store = "Компания"` (флаг `isCompanyWide = true`):

- Колонка «Итого» — показывает сумму в обычном режиме (без изменений)
- Все колонки магазинов — объединяются в одну ячейку (`colspan = stores.length`), значение = то же что в «Итого», выравнивание по центру

Для остальных статей — поведение без изменений.

### Пример

```
Обычная статья:
┌─────────────────────┬───────────┬──────────┬──────────┬──────────┐
│ Зарплата продавцов  │  500 000  │  150 000 │  100 000 │   80 000 │ ...
└─────────────────────┴───────────┴──────────┴──────────┴──────────┘

Статья «Компания»:
┌─────────────────────┬───────────┬──────────────────────────────────┐
│ Закупка товара      │18 090 000 │         18 090 000                │
│                     │           │  ← colspan = N магазинов →        │
└─────────────────────┴───────────┴──────────────────────────────────┘
```

---

## Изменения в коде

### `buildSummaryData(rows)`

**1. Исключить `"Компания"` из `stores`:**

`"Компания"` продолжает накапливаться в `storeTurnover` во время обхода строк (это нормально — `storeTurnover` используется только для сортировки). Сначала сортировка, потом фильтрация:

```js
const stores = Object.keys(storeTurnover)
  .sort((a, b) => {
    const diff = storeTurnover[b] - storeTurnover[a];
    return diff !== 0 ? diff : a.localeCompare(b, 'ru');
  })
  .filter(s => s !== 'Компания');
```

**2. Флаг `isCompanyWide` для статей:**

Определяется из `cellData[article]` (сырые данные) — **не** из `byStore` (который уже использует отфильтрованный `stores`). Условия для `isCompanyWide = true`:
- `cellData[name]` существует, И
- `cellData[name]` содержит хотя бы один ключ (не пустой объект), И
- все ключи в `cellData[name]` равны `"Компания"`

```js
const keys = Object.keys(cellData[name] || {});
const isCompanyWide = keys.length > 0 && keys.every(s => s === 'Компания');
```

Флаг вычисляется **внутри `articleNames.map(name => ...)`** — в том же месте где строится `byStore`, до возврата объекта статьи:

```js
articleNames.map(name => {
  const byStore = {};
  stores.forEach(s => { byStore[s] = (cellData[name] && cellData[name][s]) || 0; });
  const total = articleTotal[name] || 0;
  const keys = Object.keys(cellData[name] || {});
  const isCompanyWide = keys.length > 0 && keys.every(s => s === 'Компания');
  return { name, byStore, total, isCompanyWide };
});
```

### `renderSummaryMatrix()`

Переменная `stores` — та же, что возвращена из `buildSummaryData` и используется для рендера `<thead>`. Это единственный `stores` в функции.

В блоке рендера строк статей (`g.articles.forEach`):

- Если `a.isCompanyWide === true`:
  ```html
  <td colspan="${stores.length}" style="text-align:center">
    ${renderSummaryCell(a.total)}
  </td>
  ```
- Если `a.isCompanyWide === false` — рендер без изменений (по одной `<td>` на каждый store).

Дополнительно добавить проверку: если `stores.length === 0` (эта проверка гарантирует, что код рендера ячеек `colspan` никогда не выполняется при `stores.length = 0`, исключая невалидный `colspan="0"`). (данные есть, но все магазины отфильтрованы — например, месяц содержит только «Компания»-транзакции), показать сообщение вместо таблицы:
```js
if (stores.length === 0) {
  wrap.innerHTML = `<p style="padding:24px;color:#9CA3AF;text-align:center;">Нет данных по магазинам за ${escHtml(MONTH_NAMES[currentMonth - 1])}</p>`;
  return;
}
```
Эта проверка добавляется сразу после `const { stores, groups } = buildSummaryData(allRows);`.

### Строки-заголовки групп (`smg-row`)

Рендер без изменений: каждая store-колонка получает одну `<td>` с subtotal.

**Известное поведение:** `subtotals[store]` для группы суммирует только обычные статьи (у company-wide статей `byStore[s] = 0` для всех магазинов). Это означает:
- Группа со смешанными статьями: `Итого` > сумма store-колонок — разница = компанийская доля
- Группа только из company-wide статей: все store-колонки = 0, хотя `Итого` ненулевое

Это **намеренно и не требует изменений**: компанийские расходы нельзя разбить по магазинам. Разрыв между «Итого» и суммой store-колонок в заголовке группы является корректным отражением данных. Реализатор не должен пытаться «выровнять» эти числа.

---

## `toggleSummaryZeros`

Работает без изменений: логика оперирует на уровне `<tr>` (атрибут `data-zero`, `style.display`) и не зависит от структуры `<td>` внутри строки.

---

## Граничные случаи

| Случай | Поведение |
|--------|-----------|
| Статья имеет транзакции только в `"Компания"` | `isCompanyWide = true` — colspan рендер |
| Статья не имеет транзакций (нет в `cellData`) | `isCompanyWide = false` (keys.length = 0) — обычный рендер, нулевая строка |
| Статья имеет данные в `"Компания"` И в магазинах (нарушение инварианта) | `isCompanyWide = false` (защитная проверка) — обычный рендер; сумма `"Компания"` войдёт в `total` но не в store-колонки |
| Все данные месяца — только `store = "Компания"` | `stores.length === 0` после фильтрации — показывается сообщение вместо таблицы |
| Новые значения store в будущем | Фильтруется только `"Компания"`, остальные добавляются в `stores` автоматически |
